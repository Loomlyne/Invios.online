| Token | Value | Usage |

| sm | 8px | Button internal padding, form field gaps |
| md | 16px | Default element spacing, card padding |
| lg | 24px | Section padding, grid gaps |

Accent (`--accent` #ca8a04) is reserved **exclusively** for:

Accent must NOT be used for: general links, decorative borders, list separators, or status badges.

- `paid` / `accepted` → success variant (--success #24583a)
- `overpaid` → info variant
- `rejected` / `overdue` → warning variant

## Interaction Contracts

### Public Invoice Page (`/invoices/public/[shareToken]`)

1. Page loads: branded header bar renders at top. Business logo left, business name right. Background uses `primaryColor` from branding (fallback: `--accent`).
2. `DocumentStatusBadge` renders below header, above document preview. Badge is visible without scrolling.
3. `InvoicePreview` renders centered, `mode="public"`. Max-width 900px on desktop, full-width on mobile with 16px horizontal padding.
4. Sticky "Download PDF" button: fixed bottom on mobile (bottom: 16px, full-width minus 32px margins). Desktop: pinned below document preview as a block button.
5. Footer: "Powered by Invios" — 14px, `--muted` color, centered, 24px padding above page bottom.

### Public Quotation Page (`/quotations/public/[shareToken]`)

1. Identical shell to public invoice page. Same branded header, footer.
2. Status badge rendered identically.
3. `InvoicePreview` centered, `mode="public"`.
4. Below document preview: Accept/Reject buttons rendered side-by-side (desktop) or stacked (mobile).
   - Accept: primary button, accent fill (`--accent`), "Accept Quotation" label.
   - Reject: secondary/outline button, danger border (`--danger`), "Reject" label.
5. On Reject click: optional textarea slides in below buttons (CSS transition, 200ms ease-out). Textarea placeholder: "Add a note for the sender (optional)". Reject confirmation button changes label to "Confirm Rejection" in danger fill.
6. On Accept click: immediate action. No confirmation dialog. Page updates status badge to "Accepted". Success message: "Quotation accepted — the sender has been notified."
7. If quotation status is already `accepted` or `rejected`: hide Accept/Reject buttons entirely. Show read-only status badge only.
8. Download PDF: same sticky button pattern as invoice page.

### Client Portal Page (`/portal/[portalToken]`)

1. Branded header bar: same `PublicPageShell` pattern. Business logo + name + accent background.
2. Page heading below header: "Your Documents" — 20px semibold, 32px top margin.
3. Document list: each `PortalDocumentRow` shows — document number (bold, 16px), status badge, issue date (muted), total amount (semibold, right-aligned). Full row is a link to the public document page.
4. Rows separated by 1px `--border` divider. No card wrapping per row — lean table-like list.
5. Empty state (no documents): centered empty state illustration area, heading "No documents yet", body "Documents shared with you will appear here."
6. Sections: Quotations section first (heading "Quotations"), Invoices section second (heading "Invoices"). Each section only rendered if documents of that type exist.
7. No pagination — show all documents. Portal is for a single client and typical count is small.

### Bilingual Document Rendering (D-09, D-11)

Bilingual (`language="bilingual"`):
- Document container: CSS grid, 2 equal columns, 32px gap. Left column: English (LTR). Right column: Arabic (RTL, `dir="rtl"`).
- Document width: max-width 1100px (wider than standard 900px to accommodate both columns).
- Line item rows: English `description` left, Arabic `arabicDescription` right. Each cell `text-align: start` (respects dir attribute).
- Section labels (e.g., "Invoice / فاتورة"): both languages shown, separated by " / ".
- TRN: "TRN: 100XXXXXXXXX" on English side; "رقم التسجيل الضريبي: 100XXXXXXXXX" on Arabic side.

Arabic-only (`language="ar"`):
- Document container: `dir="rtl"`. All text right-aligned.
- Numbers (amounts, quantities): remain `dir="ltr"` via `unicode-bidi: isolate` on number spans.
- Font: IBM Plex Sans Arabic loaded via `next/font/google`. Applied to document container only (not root layout).
- Document width: standard 900px max-width.

## RTL and Bilingual Layout Rules

These rules are binding on the executor and auditor:

1. RTL direction (`dir="rtl"`) is scoped to the document container only. The branded header and footer remain LTR regardless of document language.
2. Arabic font (`IBM Plex Sans Arabic`) is loaded via `next/font/google` and applied only within the document container scope — not the root `<html>` element.
3. Tailwind `rtl:` variant is used for RTL-conditional utilities within the document container. The rest of the page uses standard LTR utilities.
4. Bilingual side-by-side layout collapses to stacked (English above, Arabic below) at the `md` breakpoint (768px). The stacked layout uses standard LTR for English and `dir="rtl"` scoped to the Arabic block.
5. Number spans (amounts, quantities, dates) inside RTL containers must have `unicode-bidi: isolate; direction: ltr` to prevent number reversal.
6. The `lang` attribute on the document container: `lang="en"` for English, `lang="ar"` for Arabic-only, `lang="en"` on English column and `lang="ar"` on Arabic column for bilingual.

## Slug Routing Transition (UX-04)

Visual contract only — routing is a technical concern for the executor, but UI states are defined here:

| State | Visual Behavior |
|-------|----------------|
| UUID in URL (old ID-based link) | Transparent 308 redirect — user sees no error, lands on slug URL |
| Slug alias (old slug after rename) | Transparent 301 redirect — user sees no error, lands on current slug URL |
| Unknown slug (no alias found) | Standard Next.js `notFound()` — renders 404 page |

No custom 404 UI needed in Phase 4. The existing not-found.tsx handles it.
