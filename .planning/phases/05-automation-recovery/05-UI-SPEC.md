## Interaction Contracts

### Version History Panel (AUTO-01, AUTO-02)

Surface: Invoice detail page (`/app/invoices/[slug]`), below the ExpensesTable section.

**Panel collapsed state (default):**
- Renders as a compact row: `History` label (11px uppercase muted) + chevron icon + version count badge (e.g., "3 saved versions").
- Toggle button: full-width, ghost variant, `min-h-[44px]`, `ChevronDown` icon rotates 180deg when expanded (CSS transition 200ms ease-out).
- If no versions exist (new invoice, never saved): hide panel entirely. Do not render empty panel.

**Panel expanded state:**
- Renders below the toggle row as an inline list (not a drawer, not a Sheet).
- Each version row displays:
  - Date/time: formatted as "Apr 10, 2026 · 14:32" — 14px regular muted
  - Total amount: `formatCurrency(snapshot.total, currency)` — 14px semibold foreground, right-aligned
  - "Restore" button: ghost variant, 14px, right side of row, `min-h-[44px]` touch target
- Rows separated by 1px `border-border` divider. No card wrapping per row.
- Most recent version is listed first (descending by `created_at`).
- Panel header row: "Version History" (14px semibold) + "Up to 10 saved" (11px muted) on right side.
- Max 10 rows shown (matches D-02 rolling window). No pagination — cap is the UI limit.

**Restore flow (per D-05):**
1. User clicks "Restore" on any version row.
2. `VersionRestoreDialog` opens (shadcn Dialog, not AlertDialog — need to show comparison content).
3. Dialog content:
   - Heading: "Restore this version?" — 20px semibold
   - Body: "This will update the invoice line items, amounts, and dates to match the saved version. Existing payments and expenses are preserved."
   - Comparison block:
     - "Current total" label → current invoice total (14px muted label, 16px semibold value)
     - Arrow icon (`MoveRight`) in muted
     - "Restored total" label → snapshot total (14px muted label, 16px semibold value in accent-strong if different, foreground if same)
   - Warning line (only if payments exist): "Payment status will recalculate based on the restored total." — 13px muted, italic.
   - Actions row: "Cancel" (ghost button) + "Restore Version" (destructive variant — `bg-danger text-on-dark`)
4. On confirm: `restoreInvoiceVersionAction` is called. Dialog closes. Page revalidates (server action calls `revalidatePath`). Success: inline success toast or message "Invoice restored to selected version." below the panel toggle.
5. On cancel: dialog closes, no change.

**Mobile behavior:**
- Panel toggle and rows are full-width. Same visual, same interaction.
- "Restore" button stacks below the date/total on screens below `md` breakpoint (768px) if horizontal space is insufficient. Alternatively, use `min-w-[44px]` icon-only restore button on mobile (clock-return icon with aria-label="Restore this version").

### Recurring Billing Controls (AUTO-03)

Two surfaces. Both use `RecurringConfigForm`.

#### Surface A: Invoice Builder (new invoice creation)

Placement: Below the "Notes & terms" section in `DocumentBuilder`, before the submit button. Rendered only when `kind === "invoice"`.

**Toggle row:**
- Label: "Repeat this invoice" — 14px regular
- Toggle: shadcn Switch component (or equivalent pill toggle using CSS). When off: no config fields shown. When on: frequency + next date fields slide in below (CSS transition, 200ms ease-out, `max-height` animation or conditional render).
- The toggle is rendered as a horizontal row: `flex items-center justify-between gap-3`.

**Config fields (visible when toggle is on):**
- "Frequency" row: three pill buttons side-by-side — "Weekly", "Monthly", "Quarterly".
  - Pill style: `rounded-full border border-border px-4 h-10 text-sm font-medium`. Selected: `bg-foreground text-on-dark border-foreground`. Unselected: `bg-surface text-foreground`.
  - Only one can be selected at a time. Default: "Monthly".
- "Next generation date" row: `DatePicker` component. Label: "Next generation date". Helper text below field: "The next draft will be created on this date." — 12px muted.
  - Default value: auto-calculated from issue date + frequency interval (calculated in the component, not user-entered unless overridden).
- These fields submit as hidden inputs or are serialized into the invoice creation form payload.

**Form submission:** The recurring config is submitted alongside the main invoice creation. No separate save step.

#### Surface B: Existing Invoice Detail Page

Placement: Within the main invoice detail `Card`, in the actions row alongside Edit / Status / Export buttons. Renders as a single "Make recurring" `Button` with `RefreshCw` icon.

- Button variant: `secondary`, size `sm`. Label: "Make recurring".
- If invoice already has an active recurring schedule: button label changes to "Recurring: Monthly" (or whichever frequency). Clicking opens the config dialog to edit or cancel the schedule.
- Clicking "Make recurring" opens a `Dialog` containing the `RecurringConfigForm` (same fields as Surface A: frequency pills + next generation date).
- Dialog heading: "Set up recurring billing" — 20px semibold.
- Dialog actions: "Cancel" (ghost) + "Save schedule" (primary accent-fill button).
- After save: `revalidatePath` updates the page. The button label reflects the active schedule.

**Recurring badge on invoice list (Claude's discretion):**
- Add a `Badge` variant="outline" with `RefreshCw` icon (12px) + "Recurring" label to invoice list rows where a recurring schedule is active.
- Badge renders inline after the invoice number badge. 11px text, border-border style.
- This is a read-only indicator — clicking it does not open a config dialog from the list view.
