## System Context

Invios is a Next.js 15 App Router monolith with Supabase Postgres, server actions, and Vercel cron. v2.0 adds **operator intelligence layers** on top of existing billing entities without splitting into microservices.

```
┌─────────────────────────────────────────────────────────────┐
│                     Next.js App Router                       │
├──────────────┬──────────────┬──────────────┬──────────────┤
│  /app/*      │ /portal/*    │ /invoices/   │ /api/cron/*  │
│  operator    │ client view  │ public/*     │ automation   │
└──────┬───────┴──────┬───────┴──────┬───────┴──────┬───────┘
       │              │              │              │
       ▼              ▼              ▼              ▼
┌─────────────────────────────────────────────────────────────┐

│  clients · invoices · quotations · payments · expenses       │

       │                              │
       ▼                              ▼
  Resend (email)              Vercel AI SDK (Phase 20)
```

## New Data Surfaces

### 1. Client Portal v2 (`portal_slug`)

```sql
-- Migration: add portal_slug to clients
ALTER TABLE clients ADD COLUMN portal_slug text;
CREATE UNIQUE INDEX clients_user_portal_slug ON clients (user_id, portal_slug);
```

- Route: `src/app/portal/[portalSlug]/page.tsx` (replace or alias `[portalToken]`)
- Lookup: `getClientByPortalSlug(slug)` with `user_id` not exposed publicly — slug globally unique per operator OR use composite public key `/{businessSlug}/{portalSlug}` if cross-tenant collision is a concern. **Recommendation:** unique per `(user_id, portal_slug)`; public URL includes operator subdomain later; for v2.0 use `/portal/[portalSlug]` with slug uniqueness enforced per account (operators rarely share slug namespace).

**Clarification:** Since portal is client-facing, slug must be unique within the operator's clients only. Two different Invios users can both have client `acme`. Public URL needs operator scope:

**Option A (v2.0):** `/portal/[portalToken]` stays primary; add **vanity alias** `/p/[portalSlug]` that redirects via lookup table including `user_id` from token mapping stored at link generation time.

**Option B (preferred):** `/portal/[operatorSlug]/[clientPortalSlug]` — requires `profiles.operator_slug`.

**Option C (simplest v2.0):** Store `portal_slug` unique per user; public URL `/portal/c/[portalSlug]?o=[operatorPublicId]` or embed operator id in path: `/portal/[operatorId]/[portalSlug]`.

**Decision for planning:** Phase 13 implements `/portal/[operatorSlug]/[portalSlug]` using new `profiles.public_slug` (auto from business name) + `clients.portal_slug`. Matches "URL to their name" for both parties.

### 2. Time Entries

```sql
CREATE TABLE time_entries (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id uuid NOT NULL REFERENCES auth.users(id),
  client_id uuid REFERENCES clients(id),
  description text NOT NULL,
  minutes integer NOT NULL,
  billable boolean DEFAULT true,
  invoiced_at timestamptz,
  invoice_id uuid REFERENCES invoices(id),
  entry_date date NOT NULL,
  created_at timestamptz DEFAULT now()
);
```

Server action: `convertTimeEntriesToInvoiceLines(entryIds, invoiceId | newDraft)`.

### 3. Automation Rules

```sql
CREATE TABLE automation_rules (
  id uuid PRIMARY KEY,
  user_id uuid NOT NULL,
  name text NOT NULL,
  enabled boolean DEFAULT true,
  trigger_type text NOT NULL, -- 'quotation_accepted' | 'invoice_overdue' | 'recurring_due' | ...
  trigger_config jsonb DEFAULT '{}',
  actions jsonb NOT NULL, -- [{ type: 'create_invoice' }, { type: 'send_reminder' }]
  created_at timestamptz DEFAULT now()
);
```

Cron job `evaluateAutomationRules` runs after existing reminder cron or as extension.

### 4. Client Intelligence (computed, not stored)

Service: `src/lib/client-intelligence.ts`

- `computeClientLTV(clientId)` — sum paid invoices
- `computePaymentReliability(clientId)` — avg days to pay vs terms
- `computeClientHealth(clientId)` — composite score + suggested actions

Cached in React server components; optional materialized view later.

### 5. Cash Flow Forecast

Service: `src/lib/cash-flow-forecast.ts`

Inputs: `buildRevenueTrend`, recurring schedules (next 3 runs), aging buckets, historical collection rate.  
Output: `{ day30, day60, day90, series: { date, projectedIn, projectedOut }[] }`.

### 6. Proposals

Extend `quotations` with:

```sql
ALTER TABLE quotations ADD COLUMN proposal_type text DEFAULT 'quotation';
ALTER TABLE quotations ADD COLUMN approval_metadata jsonb;
-- { acceptedAt, acceptedByName, acceptedByEmail, ip, userAgent }
```

Public page: enhanced accept flow with confirmation step + optional signature pad (canvas → storage URL).

### 7. Integrations Hub

```sql
CREATE TABLE webhook_subscriptions (
  id uuid PRIMARY KEY,
  user_id uuid NOT NULL,
  url text NOT NULL,
  events text[] NOT NULL,
  secret text NOT NULL,
  enabled boolean DEFAULT true
);
```

Dispatcher: `dispatchWebhook(userId, event, payload)` called from existing server actions post-commit.

### 8. AI Co-pilot

- Route: `src/app/api/ai/draft-document/route.ts` (or server action with streaming)
- Pattern: Vercel AI SDK `generateObject` with Zod schema matching `DocumentLineItem[]`
- Review: `src/lib/ai-invoice-review.ts` — deterministic checks first, LLM summary second
- No PII in logs; user opt-in toggle in settings (Phase 20)

## Security Notes

- Portal slugs: rate-limit lookups; no cross-tenant data leak (always filter `user_id`)
- Webhooks: HMAC signature with subscription secret
- AI: API keys server-side only; opt-in per user
- Automation rules: validate action types server-side; no arbitrary code execution
